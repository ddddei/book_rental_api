# 디자인 레퍼런스 진행 가이드

작성일: 2026-07-05
전제: [UI_DESIGN_GUIDE.md](UI_DESIGN_GUIDE.md)(v1)의 원칙 — Function First, Editorial, Calm & Friendly — 은 유지. 이 문서는 "레퍼런스를 어떻게 모으고, 어떻게 실제 디자인 결정으로 바꾸는가"의 절차서다.

---

## 1. 진행 방식 요약 (5단계, 총 1~2주)

```
① 레퍼런스 수집 (2~3일)
② 무드보드 정리 + 방향 1개 선정 (1일)
③ 디자인 토큰 확정 (1일)
④ 화면별 시안: 실제 코드로 1화면 프로토타입 (2~3일)
⑤ 확정 → 전체 화면 순차 적용 (PR 단위)
```

디자이너가 없는 프로젝트이므로 **Figma 시안 단계를 생략하고, 레퍼런스 → 토큰 → 코드 프로토타입으로 직행**하는 것을 권장한다. Tailwind라서 코드에서 실험하는 비용이 낮다.

---

## 2. ① 레퍼런스 수집 — 어디서, 무엇을

### 2.1 수집 채널

| 채널 | 용도 |
|---|---|
| [Mobbin](https://mobbin.com/) | 실제 출시된 앱/웹의 스크린샷·플로우. "실물" 레퍼런스로 최우선. 무료 열람 범위로 시작 |
| [Dribbble — library management 태그](https://dribbble.com/tags/library_management) | 도서관리 도메인 시안. 단, 컨셉 시안이 많아 "예쁘지만 못 만드는" 함정 주의 |
| [Dribbble — library dashboard](https://dribbble.com/tags/library-dashboard) | 통계/대시보드 영역 참고 |
| 실서비스 직접 사용 | **가장 중요.** Libby(도서관 대여 앱), 교보문고/알라딘 모바일, 국내 공공도서관 좌석/대출 키오스크 UI 등 실제 대여 흐름을 가진 서비스 |
| Linear, Notion, Airtable | "운영툴인데 친근함" 톤의 기준점. ERP 느낌 제거의 반례 교본 |

### 2.2 화면별로 나눠서 모은다 (전체 무드 3개 + 화면별 2~3개)

- **전체 무드/톤**: 에디토리얼 + 따뜻한 그린 계열 (기존 키워드: Warm, Editorial, Friendly)
- **Stats 카드**: 숫자 강조형 카드 레이아웃
- **대여 등록 폼**: 단계형(스텝) 폼 UX — 모바일 결제 플로우 참고가 유효
- **도서 목록**: 테이블 ↔ 카드 전환형 리스트 (모바일 카드형)
- **모바일 스캔 화면**: 카메라 오버레이 UI (배송 앱, 페이 앱의 QR 스캔 화면)

### 2.3 수집 시 기록 양식 (docs/design-refs/ 에 누적)

레퍼런스 하나당 3줄:
```
- 링크/스크린샷:
- 가져올 것: (예: 통계 카드의 큰 숫자 + 작은 라벨 위계)
- 버릴 것: (예: 다크 배경 — 우리 톤과 불일치)
```
"가져올 것"이 안 써지는 레퍼런스는 버린다. 15~20개 모으고 끝. 무한 수집 금지.

## 3. ② 무드보드 → 방향 선정

- 모은 레퍼런스를 3개 방향으로 클러스터링 (예: A. 미니멀 에디토리얼 / B. 카드 중심 소프트 / C. 데이터 밀도형)
- **운영자(실사용자) 1명 이상에게 보여주고 1개 선정** — 청년동 실무자의 취향이 곧 요구사항
- 선정 기준: 예쁜 것보다 "신규 직원이 즉시 사용 가능"(v1 가이드 목표)에 부합하는가

## 4. ③ 디자인 토큰 확정

v1 가이드의 컬러(#F5F8F5 / #0EA371 / #00B5C9)를 레퍼런스 기준으로 검증·조정 후, **Tailwind 4 `@theme`으로 globals.css에 코드화**한다. 이후 모든 컴포넌트는 토큰만 참조.

확정할 목록:
- 컬러: background / surface / primary / accent / text / secondary-text + 상태 3색(available/borrowed/overdue)
- 타이포 스케일: hero / section / body / caption 4단
- 라운딩·그림자: 카드(rounded-3xl 유지 여부), 버튼, 입력 필드
- 간격 체계: 섹션 간, 카드 내부 패딩

완료 조건: 토큰 문서(UI_DESIGN_GUIDE v2로 갱신) + globals.css 반영 PR 1개.

## 5. ④ 코드 프로토타입 (1화면)

- 대상: **메인 페이지 1개 전체** (Hero + Stats + Borrow + BookList가 다 있어서 토큰 검증에 최적)
- 별도 브랜치에서 UI만 변경 (v1 가이드의 "수정 금지" 목록 준수: 로직/API 불변)
- Vercel Preview URL로 운영자 확인 → 피드백 1회 반영 → 확정

## 6. ⑤ 전체 적용 (PR 단위)

확정된 토큰/패턴으로 순차 적용. PR 1개 = 화면/영역 1개:

1. 메인 페이지 (프로토타입 머지)
2. 모바일 최적화 패스 (V1 로드맵 TASK 05 승계 — 실기기 확인 필수)
3. 스캔 카메라 오버레이 UI
4. (2차 배포 기능 추가 시) 로그인/이력/회원관리 화면 — 같은 토큰으로 신규 제작
